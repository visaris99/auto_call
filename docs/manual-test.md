# TM 다이얼러 수동 테스트 체크리스트

대상: Windows WPF 클라이언트(`dotnet/App`)

실제 전화가 발신되므로 전용 CRM 테스트 테넌트와 테스트 전화번호를 사용한다. 운영 고객 데이터로 실행하지 않는다.

## 사전 조건

- [ ] Windows PC에 Android Platform Tools 또는 배포본의 `adb/adb.exe`가 있다.
- [ ] 테스트 단말에서 USB 디버깅을 허용했다.
- [ ] `adb devices`에 사용할 단말이 `device` 상태로 표시된다.
- [ ] 테스트 CRM 계정과 테스트 리드가 준비됐다.
- [ ] CRM에 `0019_motionless_payback.sql`이 적용되고 call-attempt API가 배포됐다.
- [ ] `dotnet run --project dotnet/App/App.csproj` 또는 Release 배포본으로 실행한다.

## 로그인

- [ ] 빈 입력 제출 시 입력 안내가 표시된다.
- [ ] 잘못된 비밀번호로 로그인에 실패해도 앱이 종료되지 않는다.
- [ ] 정상 로그인 후 상단에 소속과 사용자 이름이 표시된다.
- [ ] MFA 계정은 코드 입력 후 로그인된다.
- [ ] 초기 비밀번호 계정은 웹 비밀번호 변경 안내 후 로그인이 중단된다.

## ADB 장치 선택

- [ ] 단말이 없으면 ADB 상태가 빨간색이고 발신 버튼이 비활성화된다.
- [ ] `unauthorized` 단말만 있으면 ADB 툴팁에 해당 상태가 표시된다.
- [ ] 정상 단말이 하나면 자동 선택되고 발신 버튼이 활성화된다.
- [ ] 정상 단말이 두 개 이상이면 장치 선택 전에는 발신할 수 없다.
- [ ] 두 개 이상에서 선택한 serial이 `config.json`의 `adb_serial`에 저장된다.
- [ ] 선택한 단말만 전화가 걸리고 다른 단말에는 ADB 명령이 전달되지 않는다.
- [ ] 통화 시작 후에는 장치 선택을 바꿀 수 없다.

## 큐와 선택

- [ ] 큐가 한 번의 `/leads/queue?limit=500` 요청으로 갱신된다.
- [ ] 첫 리드가 자동 선택되고 이름, 마스킹 번호, 상태, 메모가 표시된다.
- [ ] 도래한 콜백은 우선 표시되고 나머지 리드는 서버 상대 순서를 유지한다.
- [ ] 발신 시작부터 저장 완료까지 리드와 필터를 바꿀 수 없다.
- [ ] 통화 중 60초 자동 갱신이 발생해도 고객 카드가 다른 리드로 바뀌지 않는다.

## 발신과 종료

- [ ] 로그인 직후 장치 heartbeat가 큐 조회보다 먼저 성공한다.
- [ ] F1을 빠르게 여러 번 눌러도 한 번만 발신된다.
- [ ] F1 한 번에 `AUTHORIZED` call attempt가 한 건만 생성된다.
- [ ] attempt의 `device_code`, `device_serial`, 사용자, 리드가 실제 발신과 일치한다.
- [ ] 발신 직후 F2를 빠르게 여러 번 눌러도 종료 작업은 한 번만 실행된다.
- [ ] F2는 단말이 `idle`이 된 뒤에만 결과 저장 상태로 전환한다.
- [ ] 단말 종료 확인에 실패하면 경고가 표시되고 결과를 저장할 수 없다.
- [ ] 휴대폰에서 직접 종료하면 앱이 종료 상태를 감지하고 타이머가 멈춘다.
- [ ] 통화 중 창을 닫으면 종료 및 미저장 결과 폐기 여부를 확인한다.
- [ ] 미저장 결과가 있는 상태에서 창을 닫으면 별도 경고가 표시된다.

## 결과 기록

- [ ] 발신하지 않고 F3을 누르면 먼저 발신하라는 안내가 표시된다.
- [ ] 결과 미선택 상태에서 F3을 누르면 결과 선택 안내가 표시된다.
- [ ] 통화 중 F3을 누르면 실제 통화 종료 확인 후에만 저장된다.
- [ ] F3을 빠르게 여러 번 눌러도 CRM 콜로그가 한 건만 생성된다.
- [ ] 저장 재시도에도 동일한 `attemptId`가 사용된다.
- [ ] 저장 후 attempt가 `COMPLETED`이고 `call_log_id`가 연결된다.
- [ ] 저장 후 방금 통화한 리드가 큐에 남아 있어도 같은 앱 세션에서는 다시 선택되지 않는다.
- [ ] 콜백과 예약 시간의 잘못된 `HH:MM` 입력은 저장되지 않는다.
- [ ] 결과 버튼은 CRM 지원 10종이며 숫자키 `1`부터 `9`, `0`과 일치한다.

## 정책 차단

- [ ] 번호 복사, 수동 발신, 연속 발신 UI가 표시되지 않는다.
- [ ] 야간 또는 DNC 리드는 call-attempt 승인 단계에서 차단되고 단말에 발신되지 않는다.
- [ ] 최근 heartbeat가 없거나 다른 사용자 소유 장치이면 승인이 거부된다.
- [ ] 차단된 발신은 통화 세션을 남기지 않으며 다른 리드를 다시 선택할 수 있다.

## 장애와 복구

- [ ] 결과 전송 중 CRM 연결이 끊기면 로컬 대기열에 보관되고 다음 리드로 이동한다.
- [ ] ADB intent 실행이 실패하면 서버 attempt가 `CANCELED`로 변경된다.
- [ ] 앱 재실행 후 대기 중인 리드는 큐에서 다시 선택되지 않는다.
- [ ] CRM 복구 후 대기 건이 자동 전송되고 중복 콜로그가 생기지 않는다.
- [ ] 세션 만료가 통화 중 발생하면 앱이 통화를 종료하고 결과 저장을 요청한다.
- [ ] 만료 후 F3을 누르면 결과를 로컬 대기열에 보관한 뒤 로그인 화면으로 이동한다.
- [ ] 단말 통화 종료를 확인하지 못하면 로그인 화면으로 이동하지 않고 조치를 안내한다.

## 빌드와 설정

- [ ] `dotnet build dotnet/App/App.csproj -c Release`가 경고와 오류 없이 통과한다.
- [ ] Release 배포본에서 번들 `adb.exe`를 사용해 로그인, 발신, 종료, 기록이 동작한다.
- [ ] `%APPDATA%\MilestoneDialer\config.json`에 장치 serial이 보존된다.
- [ ] 사용자 메모와 평문 전화번호가 `config.json`에 기록되지 않는다.

## 검증형 자동 업데이트

- [ ] `TM_NO_AUTOUPDATE=1`이면 버전 API 요청과 다운로드가 시작되지 않는다.
- [ ] 현재 버전 이하의 manifest이면 다운로드하지 않는다.
- [ ] 정상 RSA 서명, HTTPS 호스트, 크기, SHA-256이 모두 맞을 때만 Setup을 실행한다.
- [ ] manifest 서명 불일치 시 기존 버전으로 계속 실행한다.
- [ ] HTTP 또는 `crm.milestone-sales.xyz` 이외의 호스트를 거부한다.
- [ ] 허용 호스트에서 다른 호스트로 redirect하면 거부한다.
- [ ] `Content-Length`, 실제 바이트 수, SHA-256 중 하나라도 다르면 임시 파일을 삭제한다.
- [ ] Fiddler 등으로 다운로드를 중단해도 `.part` 파일이 남지 않는다.
- [ ] 업데이트 확인을 중복 호출해도 설치 프로세스는 한 번만 시작한다.
- [ ] 정상 2.4.1 카나리에서 무음 설치 후 앱이 재실행되고 버전이 바뀐다.
